# Готовность к синхронизации задач

Приложение подготовлено к тому, чтобы в будущем предложить пользователям **синхронизацию задач с сервером или между устройствами**. Ниже — как устроены данные и куда встраивать sync.

---

## Текущая модель данных

- **Задачи** хранятся только в `localStorage` (ключ `delo_tasks`).
- Чтение: `getTasks()` из `frontend/src/lib/storage.js`.
- Запись: `saveTasks(tasks)` из того же модуля.
- **App.jsx** не обращается к `localStorage` напрямую: только через `getTasks`/`saveTasks`. Это единая точка входа для данных — слой синхронизации можно добавить, не меняя UI.

Формат задачи (минимально нужный для sync):

```ts
{
  id: string;           // уникальный, например "t_1739265600000_abc12de"
  text: string;
  createdAt: number;    // timestamp
  forDay: string;      // "YYYY-MM-DD"
  completedAt: number | null;
  isOverdue?: boolean;
  order?: number;
  reminderAt?: number | null;
  originalForDay?: string;
}
```

Экспорт/импорт уже использует этот формат (JSON с массивом `tasks`).

## Как добавить синхронизацию

### 1. Бэкенд (уже есть заготовка)

- Папка **`app/`** — Laravel, уже есть маршруты в `app/routes/api.php`.
- Для sync достаточно добавить, например:
  - `GET /api/tasks` — вернуть задачи пользователя (после авторизации).
  - `POST /api/tasks` или `PUT /api/tasks` — принять массив задач и сохранить (merge по `id` или по стратегии «последнее изменение выигрывает»).

Авторизацию можно сделать через Laravel Sanctum (уже подключён в `api.php` для `/user`).

### 2. Фронтенд — слой синхронизации

Не меняя сигнатуры `getTasks`/`saveTasks`, можно:

- **Вариант A:** завести отдельный модуль `frontend/src/lib/sync.js` (или `api.js`):
  - при старте приложения: если пользователь «включил синхронизацию», вызвать `fetchTasksFromServer()`, смержить с `getTasks()`, записать результат через `saveTasks(tasks)` и обновить state в App (например, один раз в `useEffect` при монтировании).
  - при каждом сохранении (или по таймеру/debounce): вызывать `pushTasksToServer(getTasks())`, чтобы отправлять текущий список на сервер.

- **Вариант B:** обернуть storage в «провайдер»: например, `getTasks()` внутри проверяет «включена ли синхронизация»; если да — сначала запрос к API, merge, затем возврат. Для записи — `saveTasks()` после сохранения в localStorage вызывает отправку на сервер.

В обоих вариантах **формат задач не меняется** — сервер хранит тот же JSON-массив (или его аналог в БД). Конфликты между устройствами можно решать по `updatedAt` (при необходимости добавить поле) или «last-write-wins» по `id`.

### 3. Настройки и UX

- В настройках (например, в блоке под колокольчиком или отдельный экран «Аккаунт») добавить:
  - переключатель «Синхронизировать дела с сервером» / «Войти и синхронизировать»;
  - при первом включении: запрос логина/регистрации, затем первый `GET /api/tasks`, merge с локальными, `saveTasks` и обновление state.
- Офлайн: приложение уже работает полностью по `localStorage`; при отсутствии сети sync-модуль может откладывать отправку и повторять при появлении связи.

---

## Итог

- **Расширяемость:** данные задач проходят только через `getTasks`/`saveTasks`; бэкенд в `app/` и API готовы к добавлению маршрутов.
- **Синхронизация с сервером или другим устройством** сводится к: реализовать API на Laravel, добавить слой в `frontend/src/lib/` (sync/api), вызывать его при старте и при сохранении, не меняя формат задач и минимально трогая App.jsx.
